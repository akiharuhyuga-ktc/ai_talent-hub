# 目標の個別管理・ブラッシュアップ改善 設計仕様

- 作成日: 2026-03-31
- ステータス: 承認済み（レビュー反映済み）

## 背景

タレントハブの初回運用（目標設定）で2つの問題が判明した。

1. **ブラッシュアップ時の上書き問題**: Step7で一部目標のみブラッシュアップすると、再生成された目標だけが `currentGoals` に入り、確定時に未変更の目標が消える（例: 目標①はOK→目標②③だけ再生成→確定すると目標①が消失）
2. **確定後の個別修正不可**: 目標確定後に進捗状況に応じて個別目標を修正する手段がWebアプリ上にない

## 設計方針

- 目標をMarkdownセクション単位で個別に認識・操作する
- 既存のMarkdownファイル形式を維持し、方・保坂の確定済みデータとの互換性を保つ
- 新機能は既存コンポーネント・APIの拡張として実装する
- 実データは `data/members/` 配下（`MEMBERS_DIR = path.join(PROJECT_ROOT, 'data', 'members')`）

---

## 1. データモデル: 目標の個別管理

### パース仕様

- セクション区切り: 正規表現 `/^## 目標([①②③④⑤])[（(](.+?)[）)][：:](.+)$/m` でMarkdownを分割
- ヘッダー部: ファイル先頭から最初の `## 目標[①②③④⑤]` の直前まで（`## 目標一覧` の有無に依存しない。`# 保坂さん 目標設計（修正版）` のような余分な行もヘッダーに含む）
- 各目標セクション: `## 目標[丸数字]` の行から次の `## 目標[丸数字]` の直前まで
- フッター部: 最後の目標セクション終了後（次に現れる `## ` または `# ` で始まる見出し行）からファイル末尾まで。目標セクション内の `---` 区切り線はフッターとみなさない
- 目標が0個の場合: `{ header: 全文, goals: [], footer: "" }` を返す（エラーにしない）

### パース対象の見出しパターン例

| ファイル | パターン | パース結果 |
|---------|---------|-----------|
| 方 | `## 目標①（実行）：クロスプラットフォーム実装の品質安定化` | label=①, type=実行, title=クロスプラットフォーム... |
| 保坂 | `## 目標②（挑戦／インパクト統合）：AI活用手法の...` | label=②, type=挑戦／インパクト統合, title=AI活用... |

### 型定義（追加）

```typescript
interface SingleGoal {
  index: number        // 1, 2, 3...
  label: string        // "①", "②", "③"
  type: string         // "実行", "挑戦", "インパクト", "挑戦／インパクト統合" 等
  title: string        // "クロスプラットフォーム実装の品質安定化"
  content: string      // セクション全体のMarkdown（## 見出し行含む）
}
```

### パーサー関数（goals.ts に追加）

- `parseGoalsToSections(markdown: string): { header: string, goals: SingleGoal[], footer: string }`
  - Markdownテキストを受け取り、ヘッダー・目標配列・フッターに分割
- `mergeGoalSections(goals: SingleGoal[], footer: string): string`
  - 目標配列とフッターを結合して **本文部分のみ** 返す（ヘッダーなし）
  - 理由: 既存の保存API（`POST /api/members/[name]/goals`）がヘッダー（`# 半期目標設定`〜`## 目標一覧`）を自動付与するため、二重付与を防ぐ

### 既存データとの互換性

- 方さんのファイル: `## 目標①（実行）：〜` `## 目標②（挑戦）：〜` `## 目標③（インパクト）：〜` + フッター `## 3目標の整合確認` → パース可能
- 保坂さんのファイル: `# 保坂さん 目標設計（修正版）` はヘッダーに含む。`## 目標①（実行）：〜` `## 目標②（挑戦／インパクト統合）：〜` + フッター `## 設計サマリー（修正後）` → パース可能

## 2. Step7ブラッシュアップの改善

### 現状の問題

フィードバック入力 → AIが全目標を再生成 → `currentGoals` を丸ごと置換 → 未変更目標が消失。

### 変更後の動作

チェックボックスで対象目標を選択 → 選択した目標だけAIが再生成 → 未選択の目標はそのまま保持してマージ。

### UI変更（Step7Refinement.tsx）

- 生成された目標を `parseGoalsToSections()` で分割し、目標ごとのカードで表示
- 各カードの左上にチェックボックス（デフォルト: 全選択）
- フィードバック入力欄は現状のまま1つ
- 「ブラッシュアップ」ボタン押下時、チェックされた目標のみAIに送信
- **ストリーミング中の表示**: 対象カードのみローディング表示（スピナー＋パルスアニメーション）、非対象カードはそのまま静的表示。ストリーミング完了後にマージ結果で全カードを更新

### API変更（generate/route.ts）

- リクエストボディに `targetGoalLabels: string[]` を追加（例: `["②", "③"]`）
- プロンプトへの注入:
  - 「以下の目標のみ再設計してください: ②③」
  - 「出力に含めるのは対象目標のみ。他の目標は出力しないでください」
  - フッター（整合確認テーブル）は全目標分を再生成する指示を含める

### マージロジック（フロントエンド側）

1. AIレスポンスを `parseGoalsToSections()` でパース
2. 元の目標配列のうち、対象ラベルに一致する目標をAI出力で置換
3. 非対象の目標はそのまま保持
4. `mergeGoalSections()` で本文Markdownに再結合（ヘッダーなし）
5. フッター（整合確認テーブル）はAI出力のものを採用

### refinementCount の扱い

- 現状の全体カウント方式を維持（チェックボックスの選択状態に関係なく、ブラッシュアップ実行1回で+1）
- 上限2回も変更なし

### 確定時の保存

- マージ済みの本文Markdownを既存の `POST /api/members/[name]/goals` に送信（APIがヘッダーを自動付与して保存）

## 3. 目標確定後の個別編集機能

### 概要

目標タブ（GoalsTab.tsx）に、確定済み目標の個別編集機能を追加する。各目標セクションに「編集」（手動）と「AIで修正」の2つの操作を提供する。

### UI変更（GoalsTab.tsx）

- 確定済み目標を `parseGoalsToSections()` で個別カード表示
- 各カード右上に2つのボタン:
  - **「編集」**: 手動編集モード
  - **「AIで修正」**: AI支援修正モード
- ウィザードが開いている間は編集ボタンを非活性にする（排他制御）

### 手動編集フロー

1. 「編集」クリック → 該当目標のMarkdownがテキストエリアに切り替わる
2. ユーザーが直接編集
3. 「保存」クリック → `mergeGoalSections()` で本文結合 → `POST /api/members/[name]/goals` で保存（APIがヘッダー付与）
4. 「キャンセル」で元に戻る

### AI修正フロー

1. 「AIで修正」クリック → 修正意図の入力欄がインラインで表示（例: 「進捗を踏まえて達成基準を引き上げたい」）
2. 送信 → 該当目標 + 修正意図を新規APIに送信 → AIがストリーミングで再生成
3. 結果をプレビュー表示（修正前: 上 / 修正後: 下 の比較表示）
4. 「採用」→ マージ → API保存 / 「やり直し」→ 入力欄に戻る
5. 「キャンセル」で修正モード自体を閉じる

### 新規APIルート

`POST /api/members/[name]/goals/edit`

リクエスト:
```typescript
{
  goal: SingleGoal,            // 修正対象の目標
  instruction: string,         // 修正意図テキスト
  memberContext: string,       // メンバープロフィール
  orgPolicy: string,           // 部方針
  evaluationCriteria: string,  // 評価基準
  allGoals: string             // 全目標のMarkdown（他の目標との整合性・重複回避用）
}
```

レスポンス: SSEストリーム（修正後の目標セクションMarkdown）

プロンプト設計:
- 「以下の目標を修正指示に従って書き換えてください」
- 「出力フォーマット（目標文・達成した姿・検証方法・中間確認・根拠）は維持すること」
- 「指示されていない項目は変更しないこと」
- 「他の目標（参考として提示）との重複を避けること」

### 保存

- 既存の `POST /api/members/[name]/goals` をそのまま利用
- 修正対象の目標だけ差し替え、他はそのまま保持してマージした本文Markdownを送信（APIがヘッダー付与）

### 作成日の扱い

- 保存APIが毎回 `new Date()` で作成日を書き込む既存挙動を維持（個別編集時も更新される＝最終更新日として機能）

### 既存データへの影響

- 方・保坂の確定済み目標ファイル: **変更なし**（パーサーが既存フォーマットに対応）
- 保存API（`POST /api/members/[name]/goals`）: **変更なし**（入力フォーマットは同じ）

### 注意: 保存APIとヘッダーの関係

既存の保存APIは `content` に対してヘッダー（`# 半期目標設定`〜`## 目標一覧`）を自動付与する。そのため `mergeGoalSections()` はヘッダーを含まない本文のみを返す設計とし、保存API経由で正しくヘッダーが1つだけ付与されるようにする。
