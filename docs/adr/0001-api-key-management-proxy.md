# ADR-0001: Claude API キー管理 — プロキシサーバー方式の採用

## ステータス

proposed

## コンテキスト

本システムは Claude API（Azure AI Foundry 経由）を使用してAI機能を提供する。
現在、APIキーは `.env.local` の環境変数で管理しているが、以下の課題がある。

### 課題

1. **スタンドアロン配布との相性が悪い**
   - 本システムは非技術者にバイナリ（ZIP）で配布して使ってもらう想定（PR #18）
   - 利用者それぞれに APIキーを取得・設定してもらうのは現実的ではない

2. **APIキーの漏洩リスク**
   - バイナリに埋め込めば全利用者に同じキーが渡る
   - 環境変数で管理するにしても、非技術者への説明コストが高い

3. **利用者の特定ができない**
   - 現在の仕組みでは誰がAPIを呼んだか追跡できない
   - コスト管理・不正利用の検知が困難

## 検討した選択肢

### 選択肢A: 利用者ごとにAPIキーを配布

- メリット: シンプル。追加インフラ不要
- デメリット:
  - 非技術者にキー設定を求めるのが非現実的
  - キーの回収・ローテーションが困難
  - 誰がどれだけ使ったか追跡しにくい

### 選択肢B: プロキシサーバー + Entra ID 認証

- メリット:
  - 利用者はAPIキーを意識しない（社内SSOでログインするだけ）
  - APIキーはプロキシサーバーのみが保持（漏洩リスク最小化）
  - JWT から利用者を特定できる（監査・コスト配分が可能）
  - 許可された人だけがAPI利用可能（アクセス制御）
  - 会社の既存 Entra ID 基盤をそのまま活用
- デメリット:
  - プロキシサーバーの構築・運用が必要
  - ネットワーク経由のため、オフライン利用不可
  - 追加のインフラコスト

### 選択肢C: バイナリにAPIキーを埋め込み + 難読化

- メリット: オフライン動作可能。インフラ追加なし
- デメリット:
  - リバースエンジニアリングでキーが抽出される
  - 全利用者が同一キーを共有（漏洩時の影響大）
  - 利用者の特定不可能

## 決定

**選択肢B: プロキシサーバー + Entra ID 認証** を採用する。

## 理由

1. **スタンドアロン配布の UX を損なわない**
   - 利用者はアプリを起動 → Entra ID でログイン → すぐ使える
   - APIキーの設定は一切不要

2. **セキュリティ要件を満たす**
   - 人事データを扱うシステムとして、利用者認証は必須（HR原則8: データセキュリティ）
   - APIキーの一元管理で漏洩リスクを最小化
   - JWT による利用者特定で監査ログが取れる

3. **既存インフラとの整合性**
   - Azure AI Foundry を使用中であり、Entra ID との統合は自然
   - 会社の Entra ID テナントがそのまま使える

## アーキテクチャ

```
┌─────────────────┐     JWT      ┌──────────────┐   API Key    ┌──────────────────┐
│  スタンドアロン  │ ──────────→ │  プロキシ     │ ──────────→ │  Azure AI Foundry │
│  アプリ (SPA)   │ ←────────── │  サーバー     │ ←────────── │  (Claude API)     │
└─────────────────┘   AIレスポンス └──────────────┘             └──────────────────┘
        │                              │
        │   Entra ID 認証              │ JWT検証
        ↓                              │ 利用者特定
┌─────────────────┐                    │ アクセス制御
│  Microsoft      │ ←─────────────────┘ 監査ログ
│  Entra ID       │
└─────────────────┘
```

### 認証フロー

1. ユーザーがアプリを起動
2. Entra ID でログイン（社内SSO）
3. JWT が発行される
4. アプリは JWT を付けてプロキシ API にリクエスト
5. プロキシが JWT を検証 → 許可されたユーザーか確認
6. プロキシが自身の API キーで Azure AI Foundry にリクエスト
7. レスポンスをアプリに返却

## 影響

- プロキシサーバーのインフラ構築が必要（Azure Functions or App Service 想定）
- アプリはオンライン前提になる（AI機能利用時）
- `.env.local` の APIキー設定は開発者向けのみとなり、エンドユーザーには不要になる
- 今後のAPI追加時は、プロキシを経由するルーティングを設計する必要がある
