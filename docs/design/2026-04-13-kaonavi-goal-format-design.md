# 設計書：カオナビ目標フォーマット対応

- 作成日：2026-04-13
- ステータス：承認済み

---

## 背景・目的

タレントハブの目標設定機能は、会社の評価制度（カオナビ スマートレビュー）が確立される前に実装されたため、カオナビの入力フォーマットと乖離がある。

カオナビは「① 短期成果評価_目標」「② 発揮能力評価_目標」の2つのテキストエリアで目標を管理する。タレントハブ側の目標生成・保存フォーマットをこの2軸に統一し、メンバーがカオナビに入力する際の手間をなくす。

---

## カオナビの評価制度（前提知識）

| | ① 短期成果評価 | ② 発揮能力評価 |
|---|---|---|
| 何を評価するか | 半期の目標に対する成果の達成度合い（What） | キャリアラダーに対する能力の習熟と発揮の度合い（How） |
| 評価の視点 | 短期（半期）の業績貢献 | 中長期的な成長・再現性 |
| 何に影響するか | 賞与額 | 昇降格（基本給の改定） |

## 新しい目標フォーマット

### ファイル構造（data/members/{name}/goals/2026-h1.md）

```markdown
# 半期目標設定

- 対象期間：2026年上期（4月〜9月）
- 作成日：YYYY-MM-DD
- メンバー：{name}

## ① 短期成果評価_目標

{目標文：半期で達成する具体的な成果・KPIを記述}

└ 達成した姿：
{変化・状態で終わる記述}

└ 検証方法：
{客観的な判断基準。複数観点を①②③で列挙}

└ 中間確認（3ヶ月時点）：
{6月末時点での確認基準}

└ 根拠：
{方針・期待・本人情報との紐づけ}

---

## ② 発揮能力評価_目標

{目標文：キャリアラダーに対する能力の発揮・行動を記述}

└ 達成した姿：
{変化・状態で終わる記述}

└ 検証方法：
{客観的な判断基準}

└ 中間確認（3ヶ月時点）：
{6月末時点での確認基準}

└ 根拠：
{キャリアラダー・方針との紐づけ}
```

### 旧形式との差分

| | 旧形式 | 新形式 |
|---|---|---|
| セクション構成 | 目標①（実行）/ 目標②（挑戦）/ 目標③（インパクト） | ① 短期成果評価_目標 / ② 発揮能力評価_目標 |
| フィールド内の構造 | 目標文・達成した姿・検証方法・中間確認・根拠 | 同じ（変更なし） |
| フィールド数 | 3 | 2 |
| 各フィールドの単位 | 目標1つ | 1ブロック（番号振りなし） |
| 整合確認テーブル | あり | なし（削除） |

## 振り分け基準（短期成果 vs 発揮能力）

**判断テスト：**
> 「期末に成果物や数値が出ていれば達成と言える目標か？」
> - YES → 短期成果評価（What）
> - NO（行動・姿勢・能力の発揮を問う）→ 発揮能力評価（How）

**補足：**
- 短期成果 = 「今期限りの成果」「数値・KPI・具体的な状態の達成」
- 発揮能力 = 「再現性がある行動・能力の発揮」「キャリアラダーへの対応」

## AIプロンプト変更方針（goal-generation.ts）

### 現在の出力仕様
- 3種類の目標（実行・挑戦・インパクト）を順番に生成
- 目標ごとに独立した構造

### 新しい出力仕様
- 2フィールドをそれぞれ1つの連続したテキストブロックとして生成
- 各フィールドは「目標文 + 達成した姿 + 検証方法 + 中間確認 + 根拠」の構造を持つ
- フィールド内に番号（目標1、目標2）は振らない

### プロンプト変更のポイント
1. Step2：設計前確認に「短期成果 vs 発揮能力の振り分け基準」を追記
2. Step3：目標構成ルールを「3種類」から「2フィールド」に変更
3. 出力フォーマット指示を新形式に更新
4. 絶対禁止事項に「フィールド内で目標1、目標2と番号を振ること」を追加

## ウィザードUI変更（Step6・Step7）

### Step6：目標生成画面

**現在：** 3目標が1ストリームで流れる単一パネル

**変更後：**
```
┌─────────────────────────────────────────┐
│ ① 短期成果評価_目標               [コピー] │
│                                          │
│ （ストリーミング表示）                    │
└─────────────────────────────────────────┘

┌─────────────────────────────────────────┐
│ ② 発揮能力評価_目標               [コピー] │
│                                          │
│ （ストリーミング表示）                    │
└─────────────────────────────────────────┘
```

- 2パネル縦並び
- 各パネル右上にコピーボタン（クリップボードにコピー、コピー後「コピーしました」表示）
- ストリーミングは2フィールドを順番に流す（①完了 → ②開始）

### Step7：精緻化・壁打ち

**現在：** 3目標からチェックボックスで選択 → フィードバック → 部分再生成

**変更後：**
```
修正したいフィールドを選んでください：

[ ] ① 短期成果評価を修正したい
[ ] ② 発揮能力評価を修正したい

フィードバック：[                              ]

[再生成する]
```

- 選択されたフィールドのみを再生成
- フィードバックテキストは共通（1つの入力欄）
- 再生成後も同じUIで追加修正可能（最大2回推奨）

## 型定義の変更（types.ts）

### GoalWizardState

```typescript
// 変更前
generatedGoals: string | null    // Step6出力（3目標まとめて）
finalGoals: string | null        // Step7後の確定版

// 変更後
shortTermGoals: string | null    // ① 短期成果評価_目標（Step6〜Step7で使用）
capabilityGoals: string | null   // ② 発揮能力評価_目標（Step6〜Step7で使用）
```

`generatedGoals` と `finalGoals` は削除。Step6生成後〜Step7精緻化後を通じて `shortTermGoals` / `capabilityGoals` で管理する。

### 精緻化の対象フィールド

```typescript
export type RefinementTarget = 'shortTerm' | 'capability' | 'both'
```

## ストリーミングの実装方針

### APIは変更しない

`/api/members/[name]/goals/generate` は現在通り1リクエスト・1ストリームで返す。AIが以下の順番で出力する：

```
## ① 短期成果評価_目標

（① の内容）

---

## ② 発揮能力評価_目標

（② の内容）
```

### フロントエンドで2パネルに分割

Step6コンポーネントがストリームを受信しながら、セクションヘッダー（`## ① 短期成果評価_目標` / `## ② 発揮能力評価_目標`）を検出して振り分ける。

```typescript
// 擬似コード
if (chunk.includes('## ① 短期成果評価_目標')) {
  activePanel = 'shortTerm'
} else if (chunk.includes('## ② 発揮能力評価_目標')) {
  activePanel = 'capability'
} else {
  append chunk to activePanel
}
```

- ① パネルへの追記中は ① のカーソルが点滅
- ② ヘッダーを検出した時点で ① 完了・② 開始
- API変更不要・generate promptの出力順序で制御

## データ移行方針

### 対象メンバーの分類

| 分類 | 対象（約） | 対応 |
|---|---|---|
| 目標記載あり | 10名（方・梁・山口・ソミ・セミョン・保坂・山田(剛)・辻・竹村・杜） | バックアップ ＋ AIマージ変換 |
| テンプレートのまま | 14名（チョン・ベ・ボロナ等） | テンプレート構造を新形式に置き換えのみ |

### 移行手順

1. 全メンバーの `2026-h1.md` → `2026-h1.md.bak` にコピー
2. 目標記載ありの10名：Claude APIを使って旧形式→新形式に変換
   - 旧：目標①（実行）＋ 目標②（挑戦）→ 短期成果評価（マージ）
   - 旧：目標③（インパクト）→ 発揮能力評価
   - 整合確認テーブルは削除
3. テンプレートのまま14名：テンプレートのセクション名を新形式に置き換え
4. 変換後の内容をマネージャーが確認（スクリプト完了後にレビュー）

### スクリプト仕様（scripts/migrate-goals-to-kaonavi-format.ts）

- 実行方法：`npx ts-node scripts/migrate-goals-to-kaonavi-format.ts`
- 処理：data/members 配下を全スキャン → 分類判定 → バックアップ → 変換 → 保存
- ログ：処理結果を標準出力に表示
- エラー時：該当ファイルをスキップしてログに記録（他メンバーに影響しない）
