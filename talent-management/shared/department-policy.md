## ミッション

**受託・待ちの組織 → 価値創出・提案型の組織へ**

モバイルアプリ開発部は「KTC内の課題解決を担う組織」であると同時に、「次の価値を自ら生み出す組織」へと役割を拡張する。

## 来期の環境認識

### なぜ今、方針を転換するのか
- 来期は過去の延長線上にはない大きな節目。これまでの成功パターンがそのまま通用するフェーズではなくなった
- 「申し込みアプリ」「Unlimited」チームは解散。「販売店DXアプリ」も現行案件完了後は直近の予定なし
- **来期以降は「既存案件前提の体制」が成立しない**。案件が自然に降ってくる状況は終わりを迎える

### これからの姿勢
- 「作る人がいて、作る案件がある」状態から、**「案件そのものを生み出す必要があるフェーズ」**へ移行
- 受動的なスタンスから、能動的な価値創出へ転換する

## 部の2つの柱

### 柱①：KTC内ソリューションのアプリ化（継続領域）
各部署が抱える業務課題・顧客課題をモバイルアプリで解決する役割。これまで通り、安定的に継続実施する。
- 社内業務支援
- 販売店向けDX
- 既存アプリの改善・エンハンス

> 「解決できる課題は、引き続き我々が責任を持つ」

### 柱②：企画提案型開発（来期から本格化）
既存案件を待つのではなく、自ら企画を立ち上げる。次につながる可能性を持つPoCを高速に回す。
- 仮説起点の企画提案
- 顧客課題の発見
- 検証可能なPoCの迅速実装

> 「成功率より、挑戦回数を増やす」

**2本を同時に回すことで、モバイルアプリ開発部の価値を最大化する。**

## 体制・技術方針

### 体制変更：OS別からクロスプラットフォームへ
| これまで | 来期から |
|----------|----------|
| iOS / Android の OS別・案件別組織 | クロスプラットフォーム前提の統合体制 |
| OSごとに組織が分断 | 共通資産の最大化・横断活用 |
| 知見・資産の共有が限定的 | コラボレーションの加速 |

### 新体制の3つの柱

| 役割 | 担当領域 | 主な責務 |
|------|----------|----------|
| **Flutter** | UI / Presentation | 新規・共通UIの迅速開発、1コードでiOS・Android、高速なイテレーション |
| **KMP** | Logic / Domain | 共通ロジックの再利用、ドメイン知識の集約、既存資産からの段階的導入 |
| **Producer** | Planning / Value | 企画と顧客価値の定義、案件創出の推進、ビジネス視点でのリード |

> 技術（Flutter/KMP）× 企画（Producer）の両輪で推進する組織構造

### 開発方針の基本スタンス
- **新規アプリ**：原則としてFlutter / KMPを採用。プロジェクト特性により最適な技術を柔軟に選定
- **ネイティブ開発**：ゼロにはしない。OS固有機能や高度なパフォーマンスが必要なケースでは選択肢として維持
- **既存アプリ**：現状の体制でエンハンスを継続。大規模改修・拡張時は案件状況に応じて移行を判断

## 来期の重点テーマ：種まきと芽出しの徹底

### 標準プロセス（企画から正式案件化まで）
```
企画提案 → PoC要件定義 → PoC開発 → 部長会デモ → 正式案件化 → 予算化 → 本開発
（RESEARCH & VALIDATION PHASE）              （PRODUCTION PHASE）
```

### 現実的な前提認識
- **試行錯誤が前提**：「企画〜部長会デモ」を何度も回すことが現実
- **学習速度を重視**：一発での成功率より、試行回数と学習速度を重要視。フィードバックを即座に反映する俊敏性が必要
- **小さく早く失敗する**：失敗は学びの源泉。コストをかけずに検証し、知見を残すことに価値がある

## R&D全体計画（2026年度）

> 出典：R&D全体計画書 v1.0（2026.02.18）

### R&Dの位置づけ

本R&Dは単なる新規企画開発ではなく、**組織進化のための戦略的投資**として位置づける。

| 目的 | 内容 |
|------|------|
| **開発プロセスの再設計** | 従来のフローを見直し、AI徹底活用型の新しい開発モデルを確立する |
| **AI活用能力の組織実装** | 個人スキルではなく、組織としてのケイパビリティへ昇華させる |
| **新規プロダクト創出力の強化** | 0→1を生み出す力を底上げし、将来の競争優位を確立する |

### R&Dの3テーマ

#### テーマ①：次世代モバイルアプリ開発プロセスの構築と検証
- **目的**：AI徹底活用型開発プロセスの確立。「AIが作り、人が監修する」モデルへの転換
- **ゴール**：従来比 **30〜50%** の開発効率化（リードタイム短縮・工数削減）
- **主なアクション**：要件定義から実装までのAIパイプライン設計、コード生成・自動テストの完全統合

#### テーマ②：First Note（体験設計アプリ）
- **目的**：顧客体験価値の向上。納車という特別な瞬間を音楽で彩る感情接点のデジタル化
- **役割**：テーマ①の**プロセス検証テーマ**として活用。AI生成コードの実践投入・プロセス有効性の実証

#### テーマ③：AI小寺社長アプリ（AI人格活用）
- **目的**：AI人格活用による新たな顧客接点創出。社長の人格をAI化し顧客エンゲージメントを強化
- **役割**：**プロダクト価値検証型R&D**。技術検証だけでなく「ユーザーにとっての価値」が成立するかを検証する実験場
- **技術的挑戦**：人格再現・音声合成・リアルタイム対話

### R&D推進体制（2チーム制）

| チーム | ミッション | 担当テーマ | 主要成果物 |
|--------|-----------|-----------|-----------|
| **チームA：プロセス革新チーム** | AI前提開発モデルの確立 | ①次世代開発プロセス構築、②First Noteを題材に検証 | 新プロセス定義書、従来比較レポート（QCD）、標準プロンプト集、導入ロードマップ |
| **チームB：AIプロダクト創出チーム** | AIを活用した新しい体験価値の創出 | ③AI小寺社長アプリ | 実プロダクト、UX検証結果、事業化可能性レポート |

### R&D成功指標（KPI）

| カテゴリ | 指標 |
|----------|------|
| **プロセス系** | 開発リードタイム短縮率 / 工数削減率 / AI活用率 / レビュー工数最適化 |
| **プロダクト系** | ユーザー評価（アプリストア・NPS）/ 継続利用率 / 社内外インパクト |
| **組織系** | AI活用スキル向上度 / メンバー満足度 / 再利用可能資産の蓄積量（プロンプト・共通モジュール数） |

### R&Dへの参加マインドセット

- **アイデアを出さない ≠ 減点**：通常業務が最優先。プレッシャーを感じる必要はない
- **アイデア創出 ＝ 加点要素**：積極的な提案はボーナスポイントとして評価される
- **強制ではなく挑戦機会**：やらされる仕事ではなく、自発的なチャレンジを歓迎する
- **失敗を資産化する**：うまくいかなかったデータこそが次の成功への重要な資産。行動しないことこそが最大のリスク
