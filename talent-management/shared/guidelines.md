# 運用ガイドライン

> AIタレントマネジメントシステムの運用ルールと活用方法

---

## 1. システム概要と基本方針

### Claudeの役割定義
本システムではClaudeを「AI秘書」として位置づける。**意思決定はマネージャーが行い、Claudeはその判断を支援・補助する**立場に徹する。

| Claudeがやること | Claudeがやらないこと |
|-----------------|---------------------|
| 情報の整理・構造化 | 評価の最終判断 |
| 目標案・質問案の提案 | メンバーへの直接フィードバック |
| 記録のドラフト作成 | 個人情報の社外共有 |
| 過去記録からの示唆出し | 人事的な意思決定（昇格・異動等） |

### 情報管理の原則
- 本フォルダ内の情報は**社外秘**として扱う
- メンバー個人の情報（1on1記録・評価ドラフト）は当該メンバーのフォルダにのみ記録する
- 横断分析（全メンバー比較）は、必要な場合のみ実施し、結果は慎重に扱う

## 2. 運用サイクル

### 年間スケジュール

```
4月          5月          6月          7月          8月          9月
│            │            │            │            │            │
├── 上期目標設定 ──────────────────────────────────────────────────┤
│  └─ 4月中に設定完了                                              │
│                                                                  │
├── 月次1on1 ─┼── 月次1on1 ─┼── 月次1on1 ─┼── 月次1on1 ─┼── 月次1on1 ─┤
│                                                                  │
│                                        ├── 中間振り返り（任意）  │
│                                                                  │
│                                                     └── 上期評価・下期目標設定
```

### 月次サイクル（1on1）

| タイミング | アクション | 使用ファイル |
|-----------|-----------|-------------|
| 1on1の3日前 | Claudeに1on1準備を依頼 | `goals/`の最新目標 + 前回の`one-on-one/`記録 |
| 1on1当日 | メモを取りながら実施 | `templates/one-on-one-template.md` |
| 1on1の翌日以内 | Claudeに記録整理を依頼 | 当日のメモ → `one-on-one/YYYY-MM.md`として保存 |

### 半期サイクル（目標設定・評価）

| タイミング | アクション | 使用ファイル |
|-----------|-----------|-------------|
| 期初（4月・10月） | Claudeに目標案を依頼 | `profile.md` + `shared/department-policy.md` |
| 期初 | メンバーと目標を合意 | `goals/{期}.md`を更新 |
| 期末（9月・3月） | Claudeに評価ドラフト依頼 | 全1on1記録 + 目標達成状況 |
| 期末 | ドラフトを元に評価を確定 | `reviews/{期}.md`として保存 |

## 3. Claudeへの依頼方法

### 基本の依頼パターン

#### 目標設定の提案を依頼する場合
```
members/{メンバー名}/profile.md と
shared/department-policy.md を参照して、
{メンバー名}の2026年上期の目標案を提案してください。
```

#### 1on1準備を依頼する場合
```
members/{メンバー名}/goals/2026-h1.md と
直近の1on1記録を参照して、
今月の1on1で確認すべきヒアリング項目を提案してください。
```

#### 1on1記録の整理を依頼する場合
```
以下の1on1メモを構造化して、
members/{メンバー名}/one-on-one/2026-{月}.md の形式で整理してください。

【メモ内容をここに貼り付け】
```

#### 評価ドラフトの作成を依頼する場合
```
members/{メンバー名}/one-on-one/ の全記録と
goals/2026-h1.md を参照して、
上期評価のドラフトを事実ベースで作成してください。
```

#### 横断分析を依頼する場合
```
全メンバーの goals/2026-h1.md を参照して、
目標の難易度バランスと進捗状況を俯瞰してください。
```

### 目標立案ルール

以下のルールは目標設定の提案・レビュー時に必ず適用すること。

#### 目標作成前の必須ヒアリング（2ステップ）

目標作成を依頼された場合、**いきなり目標案を生成してはならない**。以下の2ステップで情報収集を行ってから作成すること。

**ステップ1：マネージャーへの期待ヒアリング**
- まず「**{メンバー名}さんに対する期待を教えてください**」とマネージャーに質問し、インプットを求める
- 重点テーマ・成長方向性・避けたい方向性・前期からの引き継ぎ等を確認する

**ステップ2：メンバー本人の意見ヒアリング**
- マネージャーの回答を受け取った後、「**{メンバー名}さん本人の意見・希望があれば教えてください**」と確認する
- 本人が取り組みたいテーマ、キャリア志向、課題意識等を収集する
- マネージャーが代理で入力する場合も、「本人から聞いた内容」として入力してもらう

両方のインプットが揃ってから目標案を作成する。**理由：** マネージャーの期待とメンバーの意志の両方を反映することで、目標の納得感と実行力が高まる。

#### 目標作成の品質ルール

目標案を生成する際は、**人材育成の専門家**として以下のルールを厳守すること。

**ルール1：必ず「数値」と「期限」を含めること**
- 定性的・曖昧な目標は禁止する
> NG例：「コミュニケーション力を高める」
> OK例：「月1回のチーム勉強会を主導し、第3四半期末までに3回実施する」

**ルール2：「誰にでも当てはまる目標」は絶対に禁止する**
- 「主体的に行動する」「報告・連絡・相談を徹底する」等の汎用的な目標は禁止
- 「このメンバーの名前と役割に差し替えたら違和感がある目標」のみ許可する

**ルール3：目標ごとに「達成した姿（1文）」を添えること**
> 例：「KMP勉強会を3回主導し、チームの技術標準化に貢献した状態」

**ルール4：目標の難易度は「少し背伸びすれば届く」レベルに設定すること**
- 簡単すぎる目標・現状維持目標は禁止する

**ルール5：グループ方針・マネージャーの期待・本人の希望をすべてバランスよく反映させること**
- どれか一つに偏った目標にしないこと

#### 目標の出力フォーマット

以下のフォーマットで出力すること。目標は**3〜5個を上限**とする。

```
目標①：[目標文]
　└ 達成した姿：[1文]
　└ 関連する方針・期待：[根拠を一言で]

目標②：[目標文]
　└ 達成した姿：[1文]
　└ 関連する方針・期待：[根拠を一言で]
```

#### R&D関連目標における個別プロダクト名の禁止

- R&D関連の目標を作成する際、**特定のプロダクト名（「〇〇」アプリ、「〇〇」SDK 等）を目標文に含めてはならない**
- 目標は「技術検証の完了」「PoC成果物のデモ実施」「クロスプラットフォーム基盤の構築」など、**成果の性質・レベルで定義する**
- **理由：** R&Dはスピードサイクルが速く、個別プロダクトを指した目標は短期間で陳腐化するため。目標の有効期間（半期）を通じて意味を持つ抽象度で設定する

> **OK例：** 「AI活用型プロダクトのPoC開発を完了し、事業部へのデモ提案を実施する」
> **NG例：** 「これなにAIの機能拡張を完了し、リリースする」

### 依頼時の注意点
- **具体的なファイルパスを指示する**：Claudeが参照すべき情報を明示することで精度が上がる
- **目的を添える**：「〇〇のために」と背景を伝えると、より適切な提案が返ってくる
- **出力形式を指定する**：「箇条書きで」「表形式で」など形式を指定すると使いやすい

## 5. profile.md の更新ルール

`profile.md` はメンバーの状況変化に応じて随時更新する。

### 更新が必要なタイミング
- プロジェクトのアサイン変更時
- 役割・役職の変更時（昇格・体制変更）
- スキルや強みの新たな発見があった時（1on1や評価を通じて）
- キャリア方向性の変化があった時

### 更新時の注意
- スキル・経験欄は**事実に基づいた記述**にとどめる
- 評価的な表現（「優秀な」「問題のある」など）は避ける
- 更新日をコメントや別フィールドで記録しておくと変化が追いやすい

## 6. 評価ドラフト作成時の注意事項

Claudeが生成する評価ドラフトは**あくまで補助資料**。最終的な評価判断は必ずマネージャーが行う。

### ドラフト活用の流れ
1. Claudeに評価ドラフト生成を依頼
2. ドラフトの事実確認（記録に基づいているか確認）
3. 主観的・感情的な表現がないかチェック
4. 必要に応じて加筆・修正
5. メンバーとのフィードバック面談で活用

### ドラフトで避けるべき表現
- 「〜のはずです」「〜と思われます」などの推測表現
- 「非常に優秀」「問題がある」などの主観的評価
- 1on1記録に根拠のない評価コメント

## 7. よくあるQ&A

**Q. Claudeに話しかける前に毎回ファイルを読み込ませる必要がある？**
A. Claude Codeを`talent-management/`フォルダで起動すれば、CLAUDE.mdの指示が自動的に読み込まれる。個別のメンバー情報は依頼時にファイルパスを指定することで参照される。

**Q. 1on1の内容をそのままClaudeに貼り付けても大丈夫？**
A. 機密性の高い個人情報（健康・家族・プライベートな悩みなど）は貼り付けを避け、業務に関連する内容に絞ることを推奨する。

**Q. Claudeの提案が実態と合わない場合は？**
A. profile.mdやgoals/の情報が古い・不足している可能性が高い。まず各ファイルを更新してから再依頼する。

**Q. 複数メンバーの情報を同時に扱いたい場合は？**
A. CLAUDE.mdの「横断分析を求められた場合のみ複数フォルダを参照する」方針に従い、横断分析の目的を明示した上で依頼する。
