# モバイルアプリ開発部 2026年度 組織方針

---

## 1. ミッション

**受託・待ちの組織 → 価値創出・提案型の組織へ**

モバイルアプリ開発部は「KTC内の課題解決を担う組織」であると同時に、「次の価値を自ら生み出す組織」へと役割を拡張する。

---

## 2. 環境認識

### 外部環境の変化

- 今期は過去の延長線上にはない大きな節目。これまでの成功パターンがそのまま通用するフェーズではなくなった
- 「申し込みアプリ」「Unlimited」チームは解散。「販売店DXアプリ」も現行案件完了後は直近の予定なし
- **今期以降は「既存案件前提の体制」が成立しない**。案件が自然に降ってくる状況は終わりを迎える

### 内部環境の転換

- 「作る人がいて、作る案件がある」状態から、**「案件そのものを生み出す必要があるフェーズ」**へ移行
- 受動的なスタンスから、能動的な価値創出へ転換する
- OS別組織（iOS / Android）からクロスプラットフォーム前提の統合体制へ移行

---

## 3. 重点テーマ

### テーマ①：KTC内ソリューションのアプリ化（継続領域）

**目標：** 各部署が抱える業務課題・顧客課題をモバイルアプリで解決する役割を安定的に継続する。

**具体的取り組み：**
- 社内業務支援アプリの開発・運用
- 販売店向けDXアプリの開発・改善
- 既存アプリの改善・エンハンス

> 「解決できる課題は、引き続き我々が責任を持つ」

### テーマ②：企画提案型開発（来期から本格化）

**目標：** 既存案件を待つのではなく、自ら企画を立ち上げ、次につながる可能性を持つPoCを高速に回す。

**具体的取り組み：**
- 仮説起点の企画提案
- 顧客課題の発見
- 検証可能なPoCの迅速実装
- 標準プロセス：企画提案 → PoC要件定義 → PoC開発 → 部長会デモ → 正式案件化 → 予算化 → 本開発

**前提認識：**
- 試行錯誤が前提。「企画〜部長会デモ」を何度も回すことが現実
- 一発での成功率より、試行回数と学習速度を重視
- 小さく早く失敗する。コストをかけずに検証し、知見を残すことに価値がある

> 「成功率より、挑戦回数を増やす」

### テーマ③：R&D — 次世代開発プロセスの構築と新規プロダクト創出

**目標：** AI徹底活用型の新しい開発モデルを確立し、新規プロダクト創出力を強化する。組織進化のための戦略的投資として推進する。


**R&D推進体制：**

| チーム | ミッション | 担当テーマ |
|--------|-----------|-----------|
| チームA：プロセス革新チーム | AI前提開発モデルの確立 | 次世代開発プロセス + First Note |
| チームB：AIプロダクト創出チーム | AIを活用した新しい体験価値の創出 | AI小寺社長アプリ |

**成功指標（KPI）：**

| カテゴリ | 指標 |
|----------|------|
| プロセス系 | 開発リードタイム短縮率 / 工数削減率 / AI活用率 / レビュー工数最適化 |
| プロダクト系 | ユーザー評価（アプリストア・NPS）/ 継続利用率 / 社内外インパクト |
| 組織系 | AI活用スキル向上度 / メンバー満足度 / 再利用可能資産の蓄積量 |

## 4. 行動指針

### 個人に求めること

- **自ら仮説を持つこと** — 待つのではなく、自分の考えを起点に動く
- **顧客への価値を考え抜くこと** — 技術だけでなく「誰のどんな課題を解決するか」を常に問う
- **アイデアを形にする完遂力** — 考えるだけでなく、プロトタイプを作り、検証まで持っていく

### チームに求めること

- 個の力を集結し成果を積み上げる
- 互いの知見を共有し学習速度を高める
- 組織としてのビジネス貢献を意識する

### R&Dへの参加マインドセット

- **アイデアを出さない ≠ 減点** — 通常業務が最優先。プレッシャーを感じる必要はない
- **アイデア創出 ＝ 加点要素** — 積極的な提案はボーナスポイントとして評価される
- **失敗を資産化する** — うまくいかなかったデータこそが次の成功への重要な資産

### 目指す組織文化

| これまでの常識 | NEW STANDARD |
|----------------|-------------|
| 正解を探す開発 | 問いを創る開発 |
| 「できない理由」を探す | まず小さなプロトタイプを作る |
| AIは特別な技術 | AIは日常の文房具 |
| 失敗を避ける | 失敗を資産化する |

> ⚠️ 決して「自由に遊んでいいフェーズ」ではない

## 5. チーム体制

### 開発体制：OS別からクロスプラットフォームへ

| これまで | 2026年度から |
|----------|-------------|
| iOS / Android の OS別・案件別組織 | クロスプラットフォーム前提の統合体制 |
| OSごとに組織が分断 | 共通資産の最大化・横断活用 |
| 知見・資産の共有が限定的 | コラボレーションの加速 |

### 3つの機能チーム

| チーム | 担当領域 | 主な責務 |
|--------|----------|----------|
| **Flutter** | UI / Presentation | 新規・共通UIの迅速開発、1コードでiOS・Android、高速なイテレーション |
| **KMP** | Logic / Domain | 共通ロジックの再利用、ドメイン知識の集約、既存資産からの段階的導入 |
| **Producer** | Planning / Value | 企画と顧客価値の定義、案件創出の推進、ビジネス視点でのリード |

> 技術（Flutter/KMP）× 企画（Producer）の両輪で推進する組織構造

### 開発方針

- **新規アプリ：** 原則としてFlutter / KMPを採用。プロジェクト特性により最適な技術を柔軟に選定
- **ネイティブ開発：** ゼロにはしない。OS固有機能や高度なパフォーマンスが必要なケースでは選択肢として維持
- **既存アプリ：** 現状の体制でエンハンスを継続。大規模改修・拡張時は案件状況に応じて移行を判断

## まとめ：2026年度、私たちが目指す姿

| テーマ | 内容 |
|--------|------|
| **技術 × 企画の両輪体制** | エンジニアとプロデューサーが一体となり、KTCとKINTOの次なるビジネスを共創する |
| **クロスプラットフォーム戦略** | FlutterとKMPを軸に、開発スピードとプロダクト品質を高い次元で両立させる |

| **AI活用による組織進化** | R&Dを通じてAI前提の開発モデルを確立し、組織としてのケイパビリティを底上げする |

**2本の柱（KTC内ソリューション × 企画提案型開発）を同時に回すことで、モバイルアプリ開発部の価値を最大化する。**
