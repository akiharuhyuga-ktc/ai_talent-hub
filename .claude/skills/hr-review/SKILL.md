# HR観点レビュースキル

システムの設計・コード・PR・仕様を、人事の原則・哲学に基づいてレビューし、構造化されたフィードバックを提供する。

## リファレンス

- [HR原則チェックリスト](hr-principles.md) — 11原則の詳細（レッドフラグ・あるべき姿）

## ワークフロー

### Step 1: レビュー対象の特定

ユーザーの引数に応じてレビュー対象を取得する。**引数がない場合はユーザーに確認する**。

#### 典型的な使い方（優先順）

1. **PR の URL または番号** → `gh pr view <number>` + `gh pr diff <number>` で取得
   ```
   /hr-review https://github.com/org/repo/pull/15
   /hr-review #15
   ```

2. **ファイルパス** → そのファイルを Read で取得
   ```
   /hr-review src/lib/prompts/evaluation-draft.ts
   ```

3. **コミット済みの変更** → `git diff main...HEAD` で差分を取得
   ```
   /hr-review commits
   ```

4. **テキスト（仕様・設計案）** → ユーザーが直接提供したテキストを使用

5. **引数なし** → ユーザーに「何をレビューしますか？（PR URL、ファイルパス、または仕様テキスト）」と確認する

### Step 2: HR原則ファイルの読み込み

以下のファイルを読み込み、最新の人事方針を把握する:

- `talent-management/shared/guidelines.md` — 組織の運用原則
- `talent-management/CLAUDE.md` — AI秘書としての行動指針

**注意**: これらのファイルが存在しない場合は、Step 3 のチェックリストのみで判断する。

### Step 3: HR原則チェックリストの適用

[hr-principles.md](hr-principles.md) を読み込み、11原則に照らしてレビュー対象を精査する。

各原則について **違反**・**要注意**・**合致** のいずれかを判定する。

### Step 4: 出力

レビュー結果を以下の形式で出力する:

```markdown
## HR観点レビュー結果

### チェック結果: [🟢 リスク検出なし / 🟡 確認推奨 / 🔴 リスクあり]

### 検出事項

#### [事項のタイトル]
- **原則**: [該当するHR原則名]
- **重要度**: 高 / 中 / 低
- **該当箇所**: [具体的なコード行やドキュメント箇所]
- **問題**: [何が原則に反しているか]
- **提案**: [具体的な改善案]

（検出事項が複数ある場合は繰り返す）

### 良い点（HR原則に沿った実装）
- [既にHR原則に沿っている部分を具体的に挙げる]

### まとめ
[1-2文の総括]

---
⚠️ 本レビューはAIによる一次チェックです。人事判断を代替するものではありません。
```

## このスキルの得意・不得意

### 得意（検出力が高い）
- **プロンプト・AIの指示文** — 人事哲学がテキストで表現されるため、原則との照合がしやすい
- **設計ドキュメント・仕様書** — 「AIが評価を確定する」のような方針レベルの問題を検出しやすい
- **ワークフロー定義** — 「マネージャー確認ステップがない」など、プロセスの欠落を見つけやすい

### 不得意（限界がある）
- **UIコンポーネント** — JSXコードから「この画面は心理的安全性に反する」と判断するのは難しい。見た目の印象はコードだけでは分からない
- **APIエンドポイント** — ハンドラーのコードだけで認可の十分性を判断するには文脈が必要。ただし「認可ミドルウェアがない」「全メンバー一括取得に権限チェックがない」など仕組みレベルの問題は検出可能
- **CSS・レイアウト** — ビジュアルデザインの人事的妥当性は判断できない

### 使い方の勘どころ
- このスキルは「人事的に合格」を認定するツールではなく、**リスクを検出するツール**
- 「🟢 リスク検出なし」は「問題がない」ではなく「このチェックリストでは引っかからなかった」という意味
- UI/APIの人事的妥当性は、このスキルの結果だけでなく**人事担当者との対話で補完**すること

## レビュー時の姿勢

- **建設的であること**: 問題を指摘するだけでなく、必ず具体的な改善案を提示する
- **良い点も認める**: HR原則に沿った実装を積極的に評価する
- **トーンは支援的に**: 「これは違反です」ではなく「ここをこう変えるとより良くなります」
- **日本語で出力する**
- **Step 2 で読み込んだ最新の方針がチェックリストと矛盾する場合は、最新の方針を優先する**
